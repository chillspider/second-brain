# Platform & Stack Reference — Design Spec (v1.0)

> **Proposal.** Single canonical reference for what the JetX platform is made of, what we deliberately don't use, and the rules governing future additions. Builds on [[00-architecture]] (existing canonical stack snapshot), [[jetx-platform-principles]] (the "boring tech" rule), and [[jetx-data-ownership-principles]] (the broker + envelope rules). Once Jimmy approves, the inventory promotes into [[Project Overview (JetX Tech Vision)]] under a new section; the doc itself stays here as the reference.

## 1. What this is

The platform inventory. Every component listed has a role, an owner, and a reason it's in the stack. New apps inherit this stack; they don't invent their own. New additions require justification against §7 (stack rules).

This is descriptive of *what exists today* plus prescriptive about *what's allowed tomorrow*. It is not a roadmap (that's [[[C] 3-Year Roadmap — Design Spec]]) and not an architecture (that's the next sub-project, Architectural Structure).

## 7. Stack rules — what to add, when, and why

These rules govern future additions. They derive from [[jetx-platform-principles]] (#10 boring infrastructure) and [[[C] Governing Principles — Design Spec]] Pillar F (cultural — boring tech, anyone can google).

**R1. Default to boring.** Every component must be something a new hire can google and find mature docs for. New, exciting, or experimental tech is a red flag, not a feature.

**R2. Justify against an existing choice.** Adding a new component requires arguing why an existing one can't do the job. "It's cleaner" is not a sufficient argument. "Existing X doesn't scale past Y" or "Existing X breaks principle Z" is.

**R3. One database family per role.** PostgreSQL is THE OLTP DB. Iceberg is THE lakehouse format. Adding a second of either category (e.g., MongoDB for OLTP, Delta Lake for analytics) requires a formal challenge to this rule.

**R4. No bespoke frameworks.** No in-house SDK, no custom ORM, no homegrown auth, no platform-specific build tool. (Per [[jetx-platform-principles]] #10.)

**R5. No platform abstractions over apps.** Apps talk to the stack directly. The stack doesn't try to hide itself behind wrappers. (Apps stay independent — [[jetx-platform-principles]] #1.)

**R6. Self-hostable beats SaaS unless the SaaS is justified.** Self-hosting fits the cost + control + sovereignty thesis. SaaS is allowed when the operational cost of self-hosting exceeds the SaaS cost (e.g., we don't run our own Tailscale coordination server — we use Tailscale's).

**R7. Tools are deferred until pain is real.** Add the schema registry when JSON Schema in git breaks down. Add Kafka when NATS JetStream breaks down. Add observability traces when metrics + logs aren't enough. Never speculatively. (Per [[jetx-data-ownership-principles]] §"What to build, and when".)

**R8. Adding a component requires updating this doc.** Every new addition gets a row in §2-§6 with a one-line role. Components without a documented role get audited and removed.

## 8. Anti-stack — what we deliberately don't use

Naming the rejections protects the stack as much as naming the choices.

| Rejected | Why |
|---|---|
| **Superblocks / Retool / Appsmith** | Solves wrong problem; we have one engineer + small team shipping many apps on custom infra, not non-engineers building apps |
| **ABP Framework** | Opinionated toward Blazor/Angular; fights SvelteKit; pay 100% learning curve for 40% of value |
| **Bespoke "JetX SDK" wrapping .NET** | Violates "no platform abstractions" (R5) + "boring tech" (#10) |
| **Custom ORM** | EF Core exists; building our own is innovation budget wasted |
| **MongoDB / DynamoDB / non-PG OLTP** | One OLTP family; PG17 covers our cases; secondary OLTP would fragment ops |
| **Kafka + Schema Registry + Avro** | NATS JetStream + JSON Schema in git is the right scale for us; Kafka stack adds operational cost we don't need yet |
| **No-code / low-code platform** | Engineers ship code (per [[jetx-platform-principles]] platform-does-not-provide list) |
| **Wallet / crypto / public token rails for JetX Coins** | Closed-loop only (per [[[C] Governing Principles — Design Spec]] A4) |
| **Bespoke charger hardware** | Integration only (per [[[C] Governing Principles — Design Spec]] A7) |
| **Multi-cloud abstraction layers (Crossplane, Pulumi-everywhere)** | Self-hosted infra; abstraction over abstractions has no payoff |

## 9. Planned additions — by year

These aren't in the stack today but the roadmap implies them. Listed for visibility; promotion happens when the corresponding project lands them.

### Y1 (2026)

- **Log aggregation (likely Loki)** — needed for cross-service debugging once 9 net-new modules ship
- **CDC mechanism PG → NATS** — needed for change-event emission discipline (Pillar C)
- **Feature flag system** — needed for "iterate-in-main" pattern; lightweight self-hosted (e.g., Unleash, or custom-built per app then unified)
- **Architectural lint rules** — custom CI checks per [[[C] AI-First Framework — Design Spec]] §5.2
- **IoT broker / device gateway** — for IoT Platform v1; MQTT broker (Mosquitto or VerneMQ) most likely
- **Vending payment integration** — POS / payment provider per Vending v1 launch

### Y2 (2027)

- **Multi-region deployment tooling** — when first non-current-footprint region opens
- **Multi-tenant config isolation layer** — primitives shipped in Modular Platform v2 (EOQ1 '27); productized in Y2
- **Distributed tracing (OpenTelemetry → Tempo)** — when cross-region debugging needs it
- **Service mesh** — deferred unless network policy at scale demands it (R7)
- **EV charger SCADA / management** — for Y3 charging launch prep

### Y3 (2028)

- **Partner-facing developer portal** — when first 10+ partner operators onboard
- **ML platform** — when driver-facing AI features land (Pillar E E2 domain 4 ramp); likely Ray + MLflow self-hosted
- **Edge processing runtime** — for IoT Platform v3 (OTA + edge processing)
- **Compliance + audit log aggregation** — at scale; specifics TBD by partner + region requirements

## 11. Open questions

1. **Mobile strategy.** Per-app SvelteKit PWA, native wrappers (Capacitor/Tauri), or fully native? Decision needed before CRM v2 driver-facing journeys land.
2. **CI/CD home.** GitHub Actions vs self-hosted runners vs Gitea + Drone? Affects the agent fleet's PR-to-deploy path.
3. **Log aggregation now or after Q3?** Cost of running Loki vs cost of debugging without it.
4. **Feature flag system — buy or build?** Lightweight self-built per app initially seems fine; unified layer later.
5. **IoT broker choice.** MQTT (Mosquitto/VerneMQ) likely; could NATS JetStream cover the IoT pattern as well to keep the broker count at 1?
6. **Backup tooling.** pg_dump → MinIO + restic is the natural default; verify before the Y1 spine production-grade milestone.
7. **AI agent host.** Are the JetX bots running on JetX infrastructure, on third-party cloud, or some mix? Affects R6.
